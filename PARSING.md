# 영수증 파싱 방법 (좌표 기반 표 복원 파서)

> 이 문서는 **현재 영수증 파싱 로직의 기준 문서**입니다.
> 파싱에 문제가 생기면 **먼저 이 문서를 확인**하고, 고친 내용은 이 문서에도 반영해서 함께 업그레이드합니다.
> 작성: 2026-06-13

---

## 0. 한눈에

```
사진 → (EXIF 회전보정·전처리) → Google Vision OCR
   → ① 좌표 기반 표복원 파서 (parseReceiptByCoords)
        → 검증: 품목합 == 小計(또는 合計)?
            ├─ 통과 → 그대로 채택 (빠르고 무료, GPT 호출 안 함)
            └─ 실패 → ② 기존 GPT 하이브리드 파서로 폴백
```

- **핵심 철학**: 영수증은 원래 `품목 | 수량 | 금액` **표**다. OCR이 그걸 일렬 텍스트로 펴버리니, **글자 좌표(x,y)로 표를 다시 복원**한다.
- **신뢰성의 근거는 "검증 게이트"**: `품목합 == 영수증의 小計/合計`가 맞아떨어질 때만 자동 채택한다. 안 맞으면 사람이 보거나 GPT로 넘긴다. → **틀린 데이터가 조용히 통과하지 않는다.**

관련 파일:
- 운영 구현: `public/index.html` 의 `parseReceiptByCoords()` (이 문서의 알고리즘)
- 실험/디버그 프로토타입: `public/table-proto.html` (브라우저에서 사진 올려 단독 확인)
- 일괄 테스트: `_batchtest.js` (`node _batchtest.js test*.jpg` — 폴더의 test 이미지들을 한 번에 검증)

---

## 1. 왜 좌표인가 (배경)

기존엔 `fullTextAnnotation.text`(줄바꿈된 평문)만 정규식으로 추측했다. 문제:

- OCR이 멀티컬럼 영수증을 **일렬로 평탄화** → 품목명·수량·금액이 흩어짐 → 매칭이 어긋나 **품목 누락**.
- 휴대폰 사진의 **EXIF 회전**을 Vision이 무시 → 텍스트 순서가 뒤죽박죽.

Vision 응답의 `textAnnotations[]`에는 **글자마다 boundingPoly(좌표)** 가 들어있다. 이걸 쓰면:

- **같은 y(높이)** → 같은 "행" 복원
- **x(가로 위치)** → 이름/수량/금액 "열" 분리

> ⚠️ 운영 앱은 `preprocessReceiptImage()`에서 `createImageBitmap(file, {imageOrientation:'from-image'})`로 **EXIF를 이미 보정**해 Vision에 보낸다. 따라서 **운영 파서는 좌표 회전을 하지 않는다.**
> (프로토타입 `table-proto.html`은 원본 사진을 직접 받으므로 EXIF orientation을 읽어 좌표를 회전한다 — 이 차이를 기억할 것.)

## 2. 알고리즘 단계별

입력: `annotations` = `data.responses[0].textAnnotations.slice(1)` (첫 항목은 전체 텍스트라 제외)

### (1) 단어 박스 추출 — `extractWords`
각 annotation의 `boundingPoly.vertices` 4점에서 `cx, cy, left, right, top, bottom, h(높이)`를 계산. 줄바꿈 포함 토큰은 버림.

### (2) 열 경계 (이미지 크기 무관 상대좌표)
모든 단어의 좌우 끝으로 콘텐츠 폭 `W = R - L` 산출.
- `nameColX = L + 0.50*W` — 수량열의 왼쪽 경계(이름/수량 구분용)
- `priceColX = L + 0.60*W` — **금액열**: `cx > priceColX` 인 토큰만 금액 후보

> 절대 픽셀이 아니라 **비율**을 쓰므로 영수증 크기·해상도가 달라도 동작.

### (3) 금액 토큰 판정 — `isPriceTok`
`cx > priceColX` AND 금액 모양 AND `10 ≤ 값 < 1천만`.
- 금액 모양: `¥` 접두/`,`·`.`/뒤 세금접미사(`外`,`軽`,`内`,`*`) 허용. 예: `¥1,300`, `300`, `¥300外`.
- **하한 10**: `¥68` 같은 싼 품목도 인식(수량은 중앙열이라 금액열엔 안 옴).
- **상한 1천만**: 등록번호·전화번호 같은 비현실적 큰 수를 금액에서 제외.

### (4) 같은 줄 판정 폭 — `band`
`band = max(16, 글자높이중앙값 * 0.7)`.
- 인접 행과 안 섞이도록 **행 간격보다 작게** 잡는 게 핵심.
- 분리된 글자(`小`+`計`)를 한 줄로 합쳐 키워드 인식할 때(`lineText`)도 사용.

### (5) 품목 영역 경계 (topY ~ botY)
- `topY`: 헤더 앵커(`人数`/`伝票`/`テーブル`/`領収`) 중 **가장 아래** 줄 +30. 없으면 -∞.
  - `登録番号`는 영수증마다 위/아래라 **앵커에서 제외**.
- `botY`: 첫 금융줄(`小計`/`税抜`/`合計`)의 금액 위치 - band.
- 금액 후보 중 `topY < cy < botY` 인 것만 품목 가격으로 본다 → 헤더/푸터 숫자 차단.

### (6) 이름 줄 만들기 — `nameLines`
금액열 왼쪽(`cx < priceColX`)의 **비숫자** 토큰을 cy로 묶어 "이름 줄" 구성.
- 순수 숫자(상품코드·수량)·`@`·`¥`만 있는 토큰은 제외.
- `cleanName()`: `【…】` 부가표기 제거 → 앞쪽 세금마커(`外`/`軽`/`内`/`*`)·상품코드(숫자) 제거. 예: `外8 0012 揖保乃糸` → `揖保乃糸`.
- 단가줄(`(2個x @338)` 등 `個`/`@`/`(` 포함)은 `unitish`로 표시해 이름 후보에서 제외.

### (7) @단가 → 수량 역산 — `units`
`@` 포함 토큰에서 단가 추출: `@320`, `@¥2,110`, `(2点 x @¥600)` 등 ¥ 포함도 인식.
- 수량 = `금액 ÷ 단가` (나눠떨어지고 1~99일 때). **가까운 후보부터 순회**하여 나눠떨어지는 단가를 채택(위/아래 단가가 섞여 있어도 올바른 것 선택).
- OCR이 수량 숫자를 놓쳐도(예: Vision이 "4"를 못 읽음) 단가로 복구된다.

### (8) 품목 조립
금액열의 각 금액(위→아래)에 대해:
1. **이름**: 그 금액의 가장 가까운 "위쪽(또는 같은 줄)" 미사용 이름 줄(단가줄 제외, `dy ∈ [-30, 150]`, 비용=`|dy-50|`). 한 번 쓴 이름 줄은 `used` 처리.
2. 이름이 없거나 메타(`店`/`登録`/`小計` 등)면 품목 아님 → 건너뜀.
3. **수량**: 중앙열(`nameColX < cx < 금액cx`)의 작은 정수(같은 band 내). 없으면 1. @단가 역산이 있으면 그 값으로 보정/복구.

### (9) 합계 & 검증
- `小計` = 같은 줄에 `小計`가 있는 금액. `合計` = `合計`/`総額`/`お会計` 줄의 금액.
- **검증 대상** `target = 小計 || 合計` (외세는 小計, 내세는 小計=合計).
- `_valid = (target > 0) && (품목합 == target)`.

### (10) 반환 shape (운영)
```js
{ store, items:[{name, qty, price}], subtotal, tax8, tax10,
  total, totalItems, cat, date, _valid }
```
- `store`: 상단 비메타 줄 / rawText 첫 유효줄. `date`: rawText 날짜 정규식. `cat`: `autoClassify(store, items)`.

## 3. 검증 게이트 + 폴백 (운영 연동)

`callGoogleVisionAPI()` 안에서:
1. Vision `data` 수신 → `textAnnotations` 추출.
2. `parseReceiptByCoords(annotations, rawText)` 시도.
3. `result._valid === true` → **그대로 채택**(`_showDebugUI` → `showOcrResult`). GPT 호출 안 함.
4. 아니면 → 기존 `showOcrDebugHybrid()`(GPT 하이브리드, 실패 시 `parseReceiptText` 룰 폴백).

> **멀티페이지(분할 촬영)** 는 페이지마다 좌표계가 달라 단순 병합이 어려우므로 **기존 GPT 흐름 유지**(좌표 파서는 단일 사진 fast-path에만 적용).

## 4. 검증 현황 (2026-06-13, 실물 6종)

| 영수증 | 유형/특징 | 결과 | 검증 |
|---|---|---|---|
| 스시집 | 회전·2줄·외세·@단가 | 12품목 정확 | ✅ |
| 마트(文化堂) | 세금마커·상품코드·괄호단가 | 7품목 정확(+수량) | ✅ |
| 라멘집 | 내세 | 5품목 정확 | ✅ |
| 규카츠집 | 회전 안 됨·@¥단가 | 2품목 정확(+수량) | ✅ |
| 다이소 | 2줄 줄바꿈 이름·동일가 반복 | 부분 | ❌ 자동차단 |
| 오염 사진 | 화면 반사·다른 영수증 겹침 | 오염 | ❌ 자동차단 |

**깨끗한 일반 영수증 4종은 품목·수량까지 정확, 까다롭거나 오염된 2종은 검증이 차단. 틀린 데이터 통과 0건.**

## 5. 알려진 한계 (다음 업그레이드 후보)

1. **2줄 줄바꿈 이름 + 동일가 반복**(예: 다이소 `GENE ネイル`/`オリーブ` 따로 + 전부 ¥100): 이름 줄 매칭이 꼬여 일부 품목 누락. → 2줄 이름 병합과 `used` 정책 개선 필요(과하면 인접 품목명 오염되니 주의: 두더지잡기 영역).
2. **오염 사진**(반사·겹침): 떠도는 금액이 섞임 → 검증이 차단하지만 파싱 자체는 불가. 촬영 가이드(단색 바탕)로 완화.
3. **이름 중간 숫자**(예: `小鉢3種`의 `3`)가 순수숫자 토큰이라 제외되어 `小鉢種`으로 나옴(사소).
4. 멀티페이지는 좌표 파서 미적용(GPT 폴백).

## 6. 디버깅 & 업그레이드 방법

1. **프로토타입으로 재현**: `public/table-proto.html` 을 열고(dev 서버 `npm run dev` → `http://localhost:3000/table-proto.html`) 문제 사진을 업로드 → "복원 상세"에서 어느 단계가 어긋났는지 확인.
2. **일괄 회귀 테스트**: 문제 사진을 프로젝트 폴더에 `testN.jpg`로 두고 `node _batchtest.js` → 모든 케이스가 여전히 통과하는지(회귀) 확인. **한 곳을 고치면 다른 곳이 깨지기 쉬우니 항상 전체 재실행.**
3. 고친 로직은 `table-proto.html`(실험)과 `index.html`의 `parseReceiptByCoords`(운영)에 **둘 다 반영**하고, 이 문서의 해당 단계와 §4·§5도 갱신.

### 튜닝 파라미터 (한 곳에서 조정)
| 이름 | 현재값 | 의미 |
|---|---|---|
| `nameColX` | `L+0.50*W` | 이름/수량 열 경계 |
| `priceColX` | `L+0.60*W` | 금액열 시작 |
| `band` | `max(16, medH*0.7)` | 같은 줄 허용폭 |
| 금액 하한/상한 | `10` / `1e7` | 가격 인정 범위 |
| 이름 매칭 dy | `[-30, 150]`, 비용 `|dy-50|` | 금액↔이름 줄 거리 |
| @단가 수량 | `1~99`, 나눠떨어짐 | 역산 채택 조건 |
